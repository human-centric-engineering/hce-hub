# f-journal — project event / journal stream

*Feature 17 on [[plan]] (the first of the **self-hosting cluster** §17→§18→§19). Binding design: [[self-hosting#1 · The journal — one stream, many views (thread 5)|self-hosting §1]] (one append-only `ProjectEvent` stream; every "log" is a filtered view). Depends on [[f-data-model]] (the `Project`/`Feature`/`Task` tables events scope to + the satellite-FK/drift recipe — shipped) and [[f-hub-capabilities]] (the capabilities that will emit auto-events + the MCP-exposure/seed/parity recipe the authored verbs follow — shipped). **Discharges** the [[f-task-sheet]] §11 **activity-timeline** deferral (the reason §11 deferred it: `AiAdminAuditLog` is admin-only; this is the Hub's own consumer-facing event source). **Unblocks [[plan#18 · `f-feature-planning`|f-feature-planning §18]]** (its lifecycle verbs emit into this stream) and, later, the [[f-morning-brief]] "overnight" + [[f-projects]] recent-activity reads. Built via the **current GitHub flow** — can't dogfood the Hub to build its own self-hoster.*

## Intent

The narrative **substrate** for making the Hub its own system of record. One fork-owned, append-only **`ProjectEvent`** stream — scoped to project / feature / task — carries two families in one table:

- **Auto-events** the existing capabilities emit on a state change (`task_created`, `task_claimed`, `help_wanted`, …) — cheap side-effects of writes that already happen; they *are* the activity timeline.
- **Authored entries** written over MCP (`decision`, `note`) — the human/agent narrative, feature- or project-scoped.

The insight ([[self-hosting]]): the `plan.md` **decisions log**, the **work-completed** log, the §11 **activity timeline**, the f-projects **recent-activity** table, and the morning-brief **overnight** section are *not four models — they are filtered views over this one stream* (query by scope + `kind`). §17 builds the stream, its writers, and the first two reads (the §11 task timeline + a project **Log** view); §18/§15 add more writers and readers with no new model.

## Reconciliation with current repo reality   (required — done first)

Verified against `main` after #59 (self-hosting design) merged, 2026-07-17 (recon over `prisma/schema/app.prisma`, `lib/projects/capabilities/{create-task,add-backlog,flag-help-wanted,claim-task}.ts`, `lib/projects/claim-task-service.ts`, `lib/orchestration/audit/admin-audit-logger.ts`, `lib/app/{capabilities,db-drift}.ts`, `lib/db/drift-probes.ts`, `prisma/seeds/app/*`, `lib/projects/index.ts`). Each finding is a decision.

- **No `ProjectEvent` model exists → build it, fork-owned, `app_*`-mapped, in `prisma/schema/app.prisma`.** The design's sketch (self-hosting §1) is the target. → **Decision:** add `model ProjectEvent` + `enum ProjectEventKind` alongside the existing Hub models, following the block's stated conventions verbatim: `app_project_event` map, **hand-written satellite FK** `actorUserId → "user"` (never a Prisma `@relation` on `User`), **`ON DELETE SET NULL`** (an event is retained shared history — the actor drops, the record stays; mirrors `leadUserId`/`declaredByUserId`), drift-probed in `lib/app/db-drift.ts`. `actorAgentId` (Sunrise `AiAgent`) carries **no FK** — cross-tier, exactly like `Project.sidekickAgentId`. `projectId` is an intra-Hub `@relation` (cascade — an event is meaningless without its project); `featureId`/`taskId` are **nullable, unconstrained scope columns** (a feature/task can be hard-deleted while its history is retained — no cascade, no FK; the scope is a soft pointer, resolved best-effort on read). Indexes: `[projectId, createdAt]` (the project log), `[featureId]`, `[taskId]`. **`createdAt` keeps `@default(now())` but is explicitly settable** so §19's import seed can backdate — Prisma lets a create override a `now()` default.
- **`createdAt` seed-backdating is real and load-bearing for §19 → assert it here.** The whole point of the one-off import (self-hosting §5) is richly *backdated* history. → **Decision:** no special column needed — `@default(now())` is overridden by passing `createdAt` in a `create`. §17's tests include a "createdAt honoured when supplied" case so §19 can rely on it.
- **The claim path already funnels through a *shared service* → emit `task_claimed` there, once, and both callers get it.** `claimTask()` (`lib/projects/claim-task-service.ts`) is called by **both** the `claim_task` capability and the `POST …/claim` route (the §11 no-drift extraction). → **Decision:** emit the auto-event **inside `claimTask()`'s existing `executeTransaction`** — the capability and the HTTP route then both journal identically, for free, no second wiring point. Same reuse insight that motivated the service.
- **Auto-events must be atomic with the state change they record (system-of-record discipline) → emit *inside* each capability's transaction, not fire-and-forget.** `logAdminAction` is deliberately fire-and-forget (observability — a dropped audit row is tolerable). A `ProjectEvent` is the Hub's **authoritative** record (§19) — a dropped event is lost history. → **Decision:** a shared writer `recordProjectEvent(tx, {...})` that takes a **transaction client**, called *within* the same `executeTransaction` as the write (`create_task`, `add_backlog`, `flag_help_wanted`, `claimTask`), so an event exists **iff** the state change committed. `record_decision`/`add_note` are trivially transactional (the event *is* the write). This is the intentional divergence from `logAdminAction`'s fire-and-forget — flagged for the owner (OQ). *(The existing `logAdminAction` calls stay — admin-audit and the member-facing journal are different audiences/scopes/retention, not redundant.)*
- **The `ProjectEventKind` enum must cover the whole design vocabulary, even though §17 only *emits* a subset.** Design §1 lists `feature_*`, `task_pr_linked`, `task_merged` kinds whose emitters don't exist until §18 (feature lifecycle) / a later `link_pr`/`complete_task`. Enums are cheap to define, costly to migrate repeatedly. → **Decision:** seed the **full enum now**; §17 emits only what its live capabilities produce — `task_created` (`create_task` + `add_backlog`), `task_claimed` (`claimTask`), `help_wanted` (`flag_help_wanted`), plus authored `decision` / `note`. §18 starts emitting the `feature_*` kinds with **no enum migration**. (HB8-style: define the design's full shape, wire what exists.)
- **Authored verbs carry free text → the capability PII recipe applies.** `record_decision`/`add_note` bodies are free-text markdown. → **Decision:** both set `processesPii = true` and override `redactProvenance` (redact the body on the durable provenance row, like `create_task` redacts its title) — **or registration throws** (the funnel-doc'd guard). Each also needs its seeded **active `AiCapability` row + `McpExposedTool` row** (MCP exposure is not automatic) and a **`*.parity.test.ts`** pinning the class `functionDefinition` against its seed (Sunrise convention). Registered in `lib/app/capabilities.ts`.
- **All reads go through the [[f-access]] funnel → a new `lib/projects/journal.ts` read module + one events endpoint.** Consistent 404-not-403; no hand-rolled membership check. → **Decision:** `getProjectEvents(userId, projectId, { taskId?, featureId?, kinds? })` (funnel → 404-not-403; scope-filtered; `fetchUsers`-enriched actor identities, null → "former member"). One `GET /api/v1/projects/[id]/events` (`withAuth`, `parseCuidParam`) serves **both** the §11 task-sheet timeline (`?taskId=`) and the project Log tab (unscoped/decisions filter). Read-only GET → inherits the automatic 100/min section cap, no per-flow sub-cap.
- **The §11 task-sheet activity timeline is a real deferral to discharge here.** `task-sheet.tsx` shipped with the activity section deferred ("coming with the sidekick"). → **Decision:** wire it to the new `?taskId=` events read — a quiet reverse-chronological timeline of the task's events (+ its feature-scoped entries). Sidekick-*notes* stay deferred to §12 (they need the agent); only the *event* timeline lights up now.
- **The project view has a `?view=plan|board` tab scaffold → add `?view=log` (don't invent a new surface).** f-projects shipped the URL-driven tab pattern; the design says "a 'Log' tab on the project." → **Decision:** a third `?view=log` tab rendering the project-scoped stream (decisions + work-completed + activity, filterable), mounted in `project-view.tsx` beside Plan/Board. Reuses the shipped scaffold; no route restructure. Hub-Home / morning-brief reads are **not** in scope (§15 reads the same stream later).

**Tier / seam hypotheses (to confirm at build):** pure leaf-app, **zero platform edits, zero upstream asks anticipated**. New fork-owned: `ProjectEvent` model + one additive migration (+ drift probe), `lib/projects/{project-event,journal}.ts` (the writer + the read), `record_decision`/`add_note` capabilities (+ two seeds + two parity tests), `GET /api/v1/projects/[id]/events`, a `?view=log` tab under `components/hub/projects/**`, and the task-sheet timeline wiring. 1. **Substrate** — a schema migration + wiring emission into existing capabilities: reviews best whole, and it's an **integration checkpoint** (land it, confirm a real claim/create now writes an event, *then* build on it).

Deliberately **not** the plan's indicative "model alone · all-writers · all-reads" split: a model-only PR is unconsumed-until-the-set-completes (the exact f-data-model HB3 lesson), so the model ships *with* its first writers.

A claim writes both. **Recommended** (flag if you'd rather the journal subsume the admin trail — it can't; the audit log is a platform contract).
