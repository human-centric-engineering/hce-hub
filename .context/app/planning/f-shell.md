# f-shell — module-composable app shell

*Feature 06 on [[plan]]. Binding design: [[design_handoff_hce_hub/README|design handoff]] "Shell / layout" (+ `shell.jsx`, `app.jsx`); [[v1-requirements#13.1 Hub UI — user-facing working surface|§13.1]], [[v1-requirements#14. Open implementation questions for the Sunrise-side conversation|§14 Q2]] (module-composable routing), [[v1-requirements#2. Thesis|§2]]/[[v1-requirements#15. Out of scope for v1, but design accordingly|§15]] (module composability). Depends on [[f-theme]] (the token layer/chrome) + [[f-access]] (`/projects` route gate + membership loaders). First heavy greenfield UI-spine piece; the frame `f-projects`/`f-plan-view`/`f-board-view`/`f-task-sheet`/`f-sidekick`/`f-morning-brief` all mount into.*

## Intent

The Hub's **consumer-surface frame**: a new route group rooted at `/`, rendering the design's **three-column shell** (240px sidebar · 1fr main+topbar · 380px sidekick), the **module registry** primitive that makes the nav composable (Projects live; Sales/Support/Knowledge visibly stubbed), and **Hub Home** (`/`). It ships the *architecture and chrome*, not per-screen content — the projects list, boards, sheet, real sidekick, and brief land in their own features and **mount into this shell**. Getting the routing/auth/composability shape right here is the point: Module 2 (and every v1 view) must be a *mount-addition*, never a shell refactor ([[v1-requirements#14. Open implementation questions for the Sunrise-side conversation|§14 Q2]]).

## Reconciliation with current repo reality   (required — done first)

Verified against `main`, 2026-07-14 (recon over `app/` route groups + layouts, `app/(protected)/layout.tsx`, `app/admin/layout.tsx`, `lib/app/admin-nav.ts`, `lib/app/surface.ts`, `proxy.ts`, `components/forms/login-form.tsx`, `lib/auth/utils` + `clear-session`, the design `shell.jsx`/`app.jsx`). Each finding is a decision.

- **Routing pattern = route group + nested layouts (owner-confirmed 2026-07-14).** `app/(hub)/layout.tsx` renders the 3-col shell and wraps `{children}`; because **App Router layouts don't remount on child navigation**, the **sidekick lives in the layout and persists** across main-column nav for free (no parallel-slot machinery). Modules are route subtrees (`app/(hub)/projects/…`); `app/(hub)/projects/[id]/layout.tsx` **nests** the project-contextual sidebar section. → **Decision:** adding a module = a route folder + one registry entry (a mount-addition). *Rejected:* parallel routes / named slots (`@main`/`@sidekick` + `default.tsx`) — only the **main** column is URL-driven (it's `{children}`); the sidebar is nav and the sidekick is a persistent client panel, so slots add `default.tsx`/wiring cost for no independent-navigation benefit v1 needs.
- **Shell scope = new `(hub)` group; account pages stay in `(protected)` (owner-confirmed) — with the composable-shell guardrail.** The 3-col shell hosts only Hub surfaces (`/`, `/projects`, `/brief`); `dashboard`/`settings`/`profile` keep their `AppHeader` chrome (already themed warm via the consumer surface), reached from the shell footer. → **Guardrail (baked in now):** the shell layout **must not assume a project/module context** — Hub Home is itself a non-project page, and the design already shows the sidekick only on home/project/intake, so the shell must render cleanly for a plain page (sidebar + topbar + main, no sidekick). Keep the **sidebar nav + breadcrumbs registry/route-driven and generic**. Consequence: a future *uniform* shell (option ii — account pages under the Hub chrome) is a **mechanical re-parent** (move `settings/`/`profile/`/`dashboard/` folders into `(hub)`, retire the `(protected)` layout), not a rewrite — `(i)` is a structural **subset** of `(ii)`; nothing built here is thrown away. (Owner-vetted the future path 2026-07-14.)
- **`/` reclaim + one group-level auth guard.** Today `/` = `app/(public)/page.tsx` (f-fork's `redirect('/dashboard')` shim, [[platform-divergences]] row 3 — which explicitly anticipated "f-shell reclaims `/`"). Hub Home at `app/(hub)/page.tsx` **collides** with it (both resolve to `/`), so the shim is **removed**. Auth: `/` can't be edge-protected via `proxy.ts`'s `startsWith` list (a `/` prefix matches everything), so **the `(hub)/layout.tsx` self-guards** with `getServerSession()` → `clearInvalidSession('/…')` (the house protected-page pattern) — one guard protects the whole group (`/`, `/projects`, `/brief`); `/projects` keeps its f-access edge gate too (defence in depth). → **Decision:** remove `app/(public)/page.tsx`; guard the group in its layout; no `proxy.ts` protected-list change for `/`.
- **Post-login lands on the Hub (`/`).** `components/forms/login-form.tsx` defaults `callbackUrl` to `/dashboard`; `proxy.ts` (~L251) bounces already-authed visitors off auth pages to `/dashboard`. For a Hub-first app the front door is Hub Home. → **Decision:** repoint both to `/` (two small keep-mine platform edits, ledgered). `/dashboard` stays a reachable page (scope i) — just no longer the default landing; retiring it is not f-shell's job.
- **Module registry mirrors the fork-owned `admin-nav.ts` seam — it is fork-owned, not a Sunrise seam.** `lib/app/admin-nav.ts` (`initAppNav()` + `registerNavSection`) is a **client-safe registry auto-wired at module load** by `components/admin/admin-sidebar.tsx`. The Hub's module concept is the Hub's own, so → **Decision:** a new fork-owned **`lib/app/hub-modules.ts`** exposes `registerHubModule({ slug, label, icon, href, status: 'active' | 'soon' })`, consumed by the shell sidebar's **Modules** section (Projects `active`; Sales/Support/Knowledge `soon`, disabled). Boundary-clean (data + `lucide-react` icons only; the `<Link>` lives in the sidebar component, honouring the `lib/app/**` no-`next/*` rule). This *is* the §2/§15 composability, expressed as data.
- **Sidekick column = shell mechanics only; content is `f-sidekick` (§12).** f-shell ships the 380px column, the topbar **toggle**, the **collapse to 2-col** grid, and a **placeholder** panel. The real chat stream + per-project agent are `f-sidekick`. Because the sidekick sits in the layout, it persists across nav (the design requirement) once f-sidekick fills it.
- **Topbar: controls now, ⌘K palette later.** Breadcrumbs derived generically from the route (the guardrail), a **⌘K trigger** (control present; the palette itself is a follow-up — the design says "wire it… not built in the prototype"), a notification **bell** (no red badge — §13.5), the sidekick toggle.
- **What f-shell does NOT build.** The **projects list + project container** (`/projects`, `/projects/[id]`) are `f-projects` (§08) — f-shell ships `/projects` as a **minimal placeholder** the Projects nav routes to, and leaves the **project-contextual sidebar section** (Board/Intake/Activity/KB + members) to `f-projects`' nested `projects/[id]/layout.tsx` (f-shell provides the *mechanism* — nested layouts contribute a sidebar section — and proves it with a throwaway stub in tests, not by building the project subtree). `/brief` is `f-morning-brief`.

**Tier / seam hypotheses (confirmed):** leaf-app. New fork-owned `app/(hub)/**` + `components/hub/**` + `lib/app/hub-modules.ts` (mirrors the `admin-nav` registry shape). **Zero core→fork seams, zero upstream asks.** **Three keep-mine platform touches** (all `/`-reclaim + landing): remove `app/(public)/page.tsx` (updates [[platform-divergences]] row 3), and repoint `login-form.tsx` + `proxy.ts` post-login landing to `/` — ledgered.

## Test strategy

Mostly **presentation + a client registry + a server auth guard** — vitest on happy-dom, no DB, mock `getServerSession`.
- **The load-bearing test (composability, §06 done-when):** register a **throwaway stub module** and assert the sidebar renders it **without any shell edit** — this is the proof that Module 2 is a mount-addition. Pairs with a `hub-modules.ts` unit test (register/list, `active` vs `soon`).
- **Breadcrumb derivation** — a pure `route/pathname → crumb[]` function (generic, per the guardrail): unit-test the Hub/Projects/{project}/{sub} cases incl. an unknown route.
- **Auth guard** — the `(hub)` layout redirects a null session to `/login` and renders children for a valid one (mock the session; the house pattern).
- **Component render** — sidebar sections (Hub/Modules/footer, active states), topbar controls present, the sidekick toggle flips the grid class. Hub Home renders.
- **No smoke** — nothing DB-/request-referential (contrast f-data-model's erasure smoke). Manual at review: browser-validate the render (warm + dim, the sidekick toggle, signed-out bounce) — the standing UI-feature DoD line ([[planning-retro]] HB6).
