# f-project-admin — project + member CRUD

*Feature 05 on [[plan]]. Binding design: [[v1-requirements#13.2 Hub admin pages — inside the existing Sunrise admin shell|§13.2]] (admin pages inside the Sunrise admin shell), [[v1-requirements#7. Multi-platform support|§7]] (host-platform stub), [[v1-requirements#10. Initial data model sketch|§10]] (Project/ProjectMember). Depends on [[f-access]] (the `ProjectMember` membership funnel this feature *writes into*) and [[f-data-model]] (the tables). **Unblocks [[plan#08 · `f-projects` — projects list + project view scaffold|f-projects §08]]** — which needs both this and f-shell.*

## Intent

The **operator surface** that gives every downstream Hub view real projects to render: create projects, manage members and roles, set the host platform, and attach a project-scoped knowledge tag — all inside the **existing Sunrise admin shell** ([[v1-requirements#13.2 Hub admin pages — inside the existing Sunrise admin shell|§13.2]]), reached through the fork-owned `admin-nav` seam. It is **admin-gated, not membership-gated** (recon #1) — an admin manages *all* projects; membership is what this feature *establishes*, and it's the consumer surface (`f-projects`) that then reads it back through the [[f-access]] funnel. The one piece of load-bearing correctness is the **lead-has-member-row invariant** carried from [[f-access]]: creating a project must also seat its lead as a `role='lead'` `ProjectMember` row, because `canAccessProject` reads `ProjectMember`, not `Project.leadUserId`.

## Reconciliation with current repo reality   (required — done first)

Verified against `main` after f-shell shipped, 2026-07-15 (recon over `lib/app/admin-nav.ts` + `lib/admin-nav/registry.ts`, `components/admin/admin-sidebar.tsx`, `app/admin/**`, `app/api/v1/admin/**`, `lib/auth/guards.ts`, `lib/projects/access.ts`, `prisma/schema/app.prisma`, the knowledge-tags route, `tests/unit/lib/app/defaults.test.ts`). Each finding is a decision.

- **The admin surface is admin-gated (`withAdminAuth`), NOT membership-gated — this feature does *not* funnel through `canAccessProject`.** `lib/projects/access.ts` is the **consumer** funnel (an ordinary user seeing only *their* projects — [[f-projects]] §08, `/projects`). The **admin** managing projects acts by admin role over *all* projects. The framing "membership-scoped" in the plan/memory is a **consumer-surface** property; it does not apply to the admin CRUD. → **Decision:** every route here is `withAdminAuth` (which also carries the `sk_` API-key fallback + automatic admin-section rate-limit via `proxy.ts`); `canAccessProject` is **not** called. This feature is the *writer* of the `ProjectMember` rows the f-access funnel later reads. *(Corollary: no anti-enumeration 404-not-403 concern here — admins are entitled to see every project; that convention is the consumer surface's.)*
- **The lead-has-member-row invariant (carried from [[f-access]]) is the load-bearing correctness.** `canAccessProject` decides membership from a `ProjectMember` row on `@@unique([projectId, userId])`; `Project.leadUserId` is a **denormalized pointer, not an access source** (confirmed in `lib/projects/access.ts` — it consults only `ProjectMember`). → **Decision:** project creation **transactionally** creates the project **and** a `role='lead'` `ProjectMember` row for the lead. The property generalises to **every** write that changes who leads: reassigning `leadUserId` moves/creates the lead's member row; a member-removal must **refuse to strip the current lead's row** (else the lead loses access to their own project); role edits keep exactly the seated lead consistent. These are the member API's correctness requirements (§4), not incidental.
- **`hostPlatform` is a plain `String`, not an enum.** `prisma/schema/app.prisma` stores `hostPlatform String`. §7: *support `sunrise` properly; stub other platforms as a descriptor + extension point, don't build them.* → **Decision:** validate against a small **fork-owned known-platform descriptor list** (`lib/projects/host-platforms.ts`: `sunrise` real; `laravel-forge`/`nextjs-other`/`none` present as **stubbed descriptors** — label + `supported:false`), Zod-restrict the input to known slugs, and store the slug. No enum, no migration. The descriptor list is the §7 extension seam.
- **`KnowledgeTag` creation is an existing service — reuse, don't rebuild; `Project.knowledgeTagId` is a plain `String` (no FK).** The tag model (`knowledge_tag`: `id/slug@unique/name/description`) is created via `prisma.knowledgeTag.create` behind `createKnowledgeTagSchema` (`app/api/v1/admin/orchestration/knowledge/tags/route.ts`). `Project.knowledgeTagId` is a nullable `String` the schema comment marks "set by f-project-admin". → **Decision:** on project create, also create a per-project `KnowledgeTag` (slug `project-<projectId>`, name = project name) in the **same transaction** and set `knowledgeTagId`. Document upload **reuses the existing Sunrise KB admin** filtered to that tag (no new capability, no new ingestion pipeline — §13.2 "set up project-scoped knowledge base" = attach a tag + point at the existing uploader). *(Split-candidate flag: if the scoped-upload turns out to need real net-new UI at build, it sheds into its own task — see Open questions.)*
- **No schema migration.** `Project`, `ProjectMember`, `ProjectStatus (planning|active|archived)`, `ProjectRole (lead|member)`, and the `knowledgeTagId`/`sidekickAgentId` columns **all shipped in [[f-data-model]]**. This feature is **pure app-layer**: fork-owned routes + pages + the nav seam-fill + a small `lib/projects/` service. No `prisma/` change → the `/pre-pr` migration-drift step is N/A.
- **Fill of `lib/app/admin-nav.ts` breaks a Sunrise "ships-empty" default test (HB2 — standing step).** `initAppNav()` ships empty; `tests/unit/lib/app/defaults.test.ts:56` asserts *"initAppNav registers no admin nav sections by default"* → `getRegisteredNavSections()` `toHaveLength(0)`. → **Standing Done-when (t-2):** adapt that assertion to expect the Hub's one section + a `platform-divergences.md` row. (`admin-nav-wiring.test.tsx` **mocks** the hook, so it is unaffected; the wiring stays proven.) Grep confirmed: the fill point is `defaults.test.ts` only.
- **Admin route home = a new fork-owned top-level area, `app/admin/projects/`** (a sibling of `app/admin/users`, `app/admin/orchestration`), with the API under `app/api/v1/admin/projects/**`. → **Decision:** *not* nested under `app/admin/orchestration/` — project administration is a distinct Hub-domain area (§13.2), and orchestration is Sunrise-owned. New folders are fork-owned and conflict-free. The nav section registered via `registerNavSection` points at `/admin/projects`.
- **The agent-config + cost-dashboard bullet (plan §05 t-5) is deferred, not built here.** Sunrise's existing agent/cost admin already lists **all** agents; "filtered to Hub agents" is only meaningful once **per-project sidekick agents exist** — which [[f-sidekick]] §12 seeds. The plan's own split note puts the sidekick-agent-seed in f-sidekick. → **Decision:** drop it as a promoted task; the existing Sunrise dashboards cover the v1 need now, and the filtering lands naturally when there are Hub agents to filter. (Recorded in Open questions + decisions log.)

**Tier / seam hypotheses (confirmed):** pure leaf-app, **zero core→fork seams, zero upstream asks**. Fills the fork-owned `lib/app/admin-nav.ts` seam (live · fork→core · fork-reachable); adds fork-owned `app/admin/projects/**` pages, `app/api/v1/admin/projects/**` routes (rate-limit automatic — admin section cap in `proxy.ts`, no handler work), and a `lib/projects/admin.ts` service. `lib/projects/access.ts` is **dormant for this feature** (it's the consumer funnel; this feature writes the rows it reads). One platform-file touch — the HB2 default-test adaptation — gets a `platform-divergences.md` row.

## Test strategy

vitest = happy-dom, no live DB ([[planning-retro]] B9): mock `@/lib/db/client`, forward `executeTransaction` to a `tx` mock; pure/schema tests import the **specific** module, not a barrel (B12).

- **The load-bearing test is the invariant in `lib/projects/admin.ts` (t-1).** Over a `tx` mock, assert `createProject` issues — **in one transaction** — the `project` create, the `role='lead'` `ProjectMember` create, and the `KnowledgeTag` create, and sets `knowledgeTagId`. Assert `reassignLead` seats the new lead's member row; assert `removeMember` **throws** when the target is the current lead; assert `setMemberRole` keeps a coherent single lead. This is the correctness carried from f-access, proven at the wiring (B27).
- **Host-platform validation** (t-1): the Zod schema accepts `sunrise`, rejects unknown slugs, and marks stubbed platforms `supported:false` — import `lib/validations/project-admin.ts` directly.
- **Route tests** (t-1): mirror the knowledge-tags route tests — `withAdminAuth` rejects non-admins; POST validates the body; writes call `logAdminAction`.
- **Component tests** (t-2): list renders, create/edit forms submit against a mocked API, member table add/remove/role behaviour; mirror existing `app/admin/**` page tests.
- **HB2** (t-2): adapt `defaults.test.ts`'s admin-nav case to expect the Hub's single registered section (a *stray* extra section still fails).
- **No new smoke** — `app:smoke:erasure` (from f-data-model) already proves the FK cascade end-to-end; the lead-member-row seating means erasing a lead CASCADEs their membership while `leadUserId` goes NULL (already modelled). No net-new DB-referential effect to prove.

  1. **Sizing: 2 PRs (API → UI)** — down from the plan's ~4 (endpoint↔UI, HB3/B25). Confirmed.
  2. **KB depth: attach the per-project tag + link to the existing Sunrise KB uploader** (no net-new upload UI). Confirmed. **Rationale + future trigger:** the per-project `KnowledgeTag` is the primitive either way, so a Hub-native project-KB surface can be added later on the *same* primitive at zero present cost.
