# f-plan-view — feature-level Plan view

*Feature 09 on [[plan]]. Binding design: [[v1-requirements#3. Human-centric principles (binding)|§3.5/§3.6]] (exploratory ordering — advisory, never enforced), design handoff "2. Project — Plan view" (`design/plan.jsx` → `PlanView`, the **default** project view). Depends on [[f-projects]] (the project-view container + `?view=plan` tab scaffold this fills — shipped) and [[f-hub-capabilities]] (the coordination reads + `computeEffectiveStatus` — shipped). Sibling of [[plan#10 · `f-board-view` — Board (Kanban)|f-board-view §10]] (shares the feature/task read substrate). **Unblocks [[plan#11 · `f-task-sheet` — task detail side sheet|f-task-sheet §11]].***

## Intent

The project's **primary planning surface**: its features in optimal working order — a *recommendation*, never enforced (§3.5/§3.6). Features sort by status band (`shipped → in_flight → planning → blocked`) then dependency depth (topological), each row expanding to its task table. This is the first surface to **render the seeded plan's real 15-node dependency graph** — the payoff of §08's `006-sample-plan` seed. It fills the Plan tab of the container [[f-projects]] shipped; [[f-board-view]] fills the Board tab next off the same read substrate.

## Reconciliation with current repo reality   (required — done first)

Verified against `main` after f-projects shipped, 2026-07-15 (recon over `lib/projects/{consumer,access,task-status,user-refs}.ts` + `lib/projects/capabilities/next-task.ts`, `components/hub/projects/{project-view,project-view-tabs,presentation}.tsx`, `app/(hub)/projects/[id]/page.tsx`, `prisma/seeds/app/006-sample-plan.ts`, `prisma/schema/app.prisma`, `design/plan.jsx`). Each finding is a decision.

- **No per-project feature/task read API exists — §09 builds it (deferred from §08, ship-nothing-unused).** `consumer.ts` `getProjectForUser` returns the header + members + feature/task **counts** only; the Hub capabilities (`next-task` etc.) read tasks but are agent-facing, differently scoped, and not a project-plan read. There is **no** feature-tree endpoint for the UI. → **Decision:** build **one enriched** `GET /api/v1/projects/[id]/plan` (`withAuth`, `parseCuidParam` on the id), funnelled through `getAccessibleProject` → **deny ≡ 404 never 403** (reuse the funnel, don't re-implement membership). It returns the whole feature tree in one request (features + dependency edges + nested tasks + resolved owner/claimer `UserRef`s + per-task effective status + per-feature progress) — **no N+1** (CLAUDE.md single-enriched-endpoint rule). The page `serverFetch`es it; *not* a loader-in-RSC.
- **`planOrder()` is a new pure function — home it in `lib/projects/`, reuse the seed's real DAG.** The design's `planOrder` (client-side, `React.useMemo`) computes dependency depth topologically and is **cycle-tolerant** (its `seen` guard returns depth 0 on a back-edge rather than looping). → **Decision:** port it to a pure `lib/projects/plan-order.ts` (mirrors `task-status.ts` — pure, unit-tested; the **load-bearing topological-correctness test**, B27). Keep it **cycle-tolerant, never cycle-rejecting** — per B26/HB4 the acyclicity *guard* is homed in [[plan#13 · `f-intake`|f-intake]]/[[plan#12 · `f-sidekick`|f-sidekick]] (the edge-among-existing-nodes writers); `planOrder()` is a *read* and must degrade gracefully on a malformed graph, never throw. It maps the DB `FeatureStatus` enum (`planning|in_flight|blocked|shipped`) → the design's band order (`shipped→in_flight→planning→blocked`), then depth. Ordering is **advisory display** (§3.5) — sorted as the default presentation, never a gate on working out of order.
- **Reuse `computeEffectiveStatus` for the task pills — Plan and Board must not diverge.** `lib/projects/task-status.ts` is the shared read primitive carrying the null-claimant + PR-blocked findings (built in f-hub-capabilities, earmarked for reuse by both boards). → **Decision:** the loader computes each task's *effective* status through it, so a claimed-by-null (erased user) or dependency-blocked task reads identically in the Plan's task table and on the §10 Board. Reuse, don't reimplement.
- **[carried-in — `f-data-model` t-3 `/code-review`] Render every nullable `user` ref gracefully — never deref it.** `ownerUserId` and `claimedByUserId` are nullable *from creation* and go NULL on erasure (SET NULL). The finding **explicitly spans `f-plan-view`** (feature-owner avatar, task-claimer avatar). → **Decision:** resolve refs with the batched `fetchUsers` (`user-refs.ts`) → `UserRef | null`; render null as "unassigned" / "former member" reusing the `presentation.ts` `initials()` + the `MemberStack` null pattern already in the container. A `user.name` on a null ref NPEs the whole view — the load-bearing null-render test rides the feature row + task row.
- **No human feature/task slug in the schema — the title carries identity; dependency chips render the depended-on feature's *title*.** The design mock assumes slugs (`f-data-model`) for the mono "feature ID" column and the "depends on f-data" chips; our `Feature`/`Task` have a cuid `id` + `title` only (no slug/key field). A cuid chip is useless. → **Decision:** drop the mono slug column — the **ordinal + title** carry identity; **dependency chips render the depended-on feature's title** (truncated); task rows show a positional **`t-N`** ordinal, not the cuid. (A human feature key is an additive schema field if ever wanted — out of scope; noted to [[futures]].)
- **No `blockedReason` field / no computed feature-blocked in v1 — render dep chips, not prose.** The design mock has `feature.blockedReason` (brick text); the schema has no such field, and feature `blocked` is a *stored* `FeatureStatus` (set by admin/intake), not derived from unmet deps in v1. → **Decision:** render the dependency chips + the stored status pill; **omit the prose blocked reason** (a computed feature-blocked is a later concern, adjacent to §10's `effectiveStatus`). Minor.
- **The container + linkable `?view=plan` tab already exist — §09 fills the plan branch only, no new route.** `ProjectView` renders a `TAB_COPY[activeTab]` placeholder; `[id]/page.tsx` fetches only the header via `getProject`. → **Decision:** fill the plan branch with a client `<PlanView>` (owns expand/collapse state); the page fetches the **active tab's** dataset (the `/plan` endpoint only when `view==='plan'` — no over-fetch, the board branch stays the §10 placeholder). The group-level auth guard + `/projects` edge gate ([[f-access]]) already protect it — **no new guard**. Task rows render but **clicking opens nothing yet** — the task sheet is [[plan#11 · `f-task-sheet`|§11]]; §09 ships the rows, §11 wires the open. **No HB2** (all files fork-owned).

**Tier / seam hypotheses (confirmed):** pure leaf-app, **zero platform edits, zero upstream asks**. New fork-owned `GET /api/v1/projects/[id]/plan` route (rate-limit automatic via `proxy.ts` session-user cap), `lib/projects/plan-order.ts` (pure) + `lib/projects/plan.ts` (loader), `components/hub/projects/plan/**`. Live reused deps: `access.ts` (`getAccessibleProject`, 404-not-403), `task-status.ts` (`computeEffectiveStatus`), `user-refs.ts` (`fetchUsers`) — none re-implemented. **No migration** (all models shipped in f-data-model → `/pre-pr` drift step N/A). **No HB2 / no `platform-divergences` row.**

## Test strategy

vitest = happy-dom, no live DB ([[planning-retro]] B9): mock `@/lib/db/client` / the funnel; components via `@testing-library/react`; `serverFetch`-mocked page smokes.

- **The load-bearing tests are `planOrder()` correctness + the membership 404-not-403 (t-1).** `plan-order.test.ts` over the **seed's real 15-node graph**: `shipped` features precede `planning`, within a band deeper-dependency features sort later, a synthetic **cycle doesn't loop or throw** (degrades to depth 0). `plan.test.ts` + the route integration test: a member gets the ordered tree; a **non-member and an unknown id are indistinguishable → both 404, never 403** (re-assert anti-enumeration at the boundary); a task with `claimedByUserId=null` reports the effective status `computeEffectiveStatus` gives (not "claimed"); a null `ownerUserId` resolves to `null`, not a throw.
- **Effective-status reuse (t-1):** assert the loader routes task status through `computeEffectiveStatus` (don't re-derive) — a dep-blocked `available` task returns `blocked`; a null-claimant `claimed` task returns to the pool.
- **UI (t-2):** `planOrder`-ordered rows render (ordinal + title + dep chips); expand toggles the task table; the progress bar reflects merged/total; a **null owner/claimer renders the fallback, never `undefined.name`**; shipped features carry reduced opacity; `serverFetch`-mocked page smoke (default Plan tab renders the seeded features). Radix/RHF-free (display surface).
