# f-hub-capabilities — Hub tools + MCP exposure

*Feature 07 on [[plan]]. Binding design: [[v1-requirements#11. Hub-specific capabilities (registered tools)|§11]], [[v1-requirements#5. Sunrise refs: `.context/orchestration/capabilities.md`, `mcp.md`, `api-keys.md`.*

## Intent

The Hub's read/write operations as **Sunrise capabilities** — the shared engine the UI, the per-project sidekick, and Claude Code (over MCP) all call through one dispatcher. Read: `next-task` (+ get-shaped reads). Write: `create-task`, `add-backlog`, `flag-help-wanted`, `claim-task`. Each is a `BaseCapability` subclass, membership-scoped through `f-access`, registered via the fork seam, seeded as an `AiCapability` row, and exposed over MCP. This is the substrate `f-sidekick`, the Plan/Board views, and `f-intake` consume — so it's built once, correct, here.

## Reconciliation with current repo reality   (required — done first)

Verified against `main` after f-access shipped, 2026-07-13, via deep recon of `lib/orchestration/capabilities/**`, `lib/orchestration/mcp/**`, `lib/auth/api-keys.ts`, the approval executor, and the audit logger. Each finding is a decision; several correct the indicative plan.

- **A capability is a `BaseCapability<TArgs,TData>` subclass** (`lib/orchestration/capabilities/base-capability.ts`): required `slug`, `functionDefinition` (OpenAI-compatible JSON schema the LLM sees), a **required** Zod `schema` (validated by the dispatcher at step 7, *before* `execute`), and `execute(args, context)` returning `this.success(...)` / `this.error(...)` — never a hand-built result. → **Decision:** mirror `built-in/user-memory.ts` (DB read/write scoped by identity) — the closest structural analog. The Zod `schema` and the `functionDefinition.parameters` JSON are hand-kept in sync per capability (as every built-in does).
- **`execute` gets `CapabilityContext.userId: string | null` + `agentId`** (`capabilities/types.ts:34-53`) — **the membership seam.** → **Decision:** every Hub tool does `if (!context.userId) return this.error('…', 'no_user_context')` (system-initiated/unauthed MCP runs have null), then calls the [[f-access]] funnel **inside `execute`**: `canAccessProject(context.userId, projectId, need)` for reads/writes, `accessibleProjectIds(context.userId)` to scope list/`next-task` queries. Use the **non-throwing** `canAccessProject` (not `requireProjectAccess`) and convert a deny to `this.error(…, 'forbidden'|'not_found')`, so the agent gets a clean tool result rather than a thrown error normalized to `execution_error`. **Not** the `guard` seam (it runs pre-validation, sees only context, and is for scope/tenant refusal — not per-resource membership).
- **Registration in code is necessary but NOT sufficient — a capability needs a seeded active `AiCapability` row to dispatch, and an `McpExposedTool` row to be MCP-visible.** `registerAppCapability(new X(), {slug?, guard?})` in `lib/app/capabilities.ts` (ships empty) wires the class; but the dispatcher dies at `capability_inactive` without an active `AiCapability` row (slug-matched), and MCP exposure requires an `McpExposedTool{ isEnabled:true }` row (`mcp/tool-registry.ts` — exposure is **not** automatic). **Per-agent binding is default-allow** (a capability with no `AiAgentCapability` row is available to all agents; a row with `isEnabled:false` opts out). → **Decision:** so **no sidekick agent is needed here** — Hub capabilities are agent-agnostic; `f-sidekick` later *scopes* them to each project's agent (opt-out) and passes `{projectId}` scope. This feature ships: the classes + the **seed** of their `AiCapability` + `McpExposedTool` rows. *(Confirm at build (B17): how Sunrise seeds the built-ins' `AiCapability` rows — a seed unit vs boot-sync — and mirror that exact path for the Hub rows. This is the one open mechanism.)*
- **Claude Code authenticates over MCP with `McpApiKey` (`smcp_` prefix), NOT `AiApiKey` (`sk_`).** Two separate key systems: `AiApiKey` (`lib/auth/api-keys.ts`, scopes `chat|analytics|knowledge|webhook|admin`) is the consumer/self-service key and is **not** what MCP uses. MCP uses `McpApiKey` (`lib/orchestration/mcp/auth.ts`, protocol scopes `tools:list|tools:execute|…`), which carries **`scopedAgentId`** and a **`scope` JSON** map threaded into `CapabilityContext.scope` (core reads no keys — an explicit fork seam). → **Decision (corrects [[plan#07 · `f-hub-capabilities` — Hub tools + MCP exposure|§07 t-6]] / [[v1-requirements#14. Open implementation questions for the Sunrise-side conversation|§14 Q6]]):** per-developer MCP access = an **`McpApiKey`** with `tools:execute`, optionally `scopedAgentId` (a project's sidekick, later) and `scope: { projectId }`. The membership check inside `execute` is the real gate; the key scope is defence-in-depth.
- **No `human_approval` gate in this feature.** Two mechanisms exist and *neither* fits a direct capability write: (a) capability `AiCapability.requiresApproval` is a **hard refuse-and-stop** (dispatcher step 6 returns `requires_approval`, `skipFollowup`, handler never runs — no pending record, no resume); (b) resumable approval is a **paused `AiWorkflowExecution`** via the workflow `human_approval` step (`engine/executors/human-approval.ts` + `approval-actions.ts`) — there is no `AiApprovalRequest` model. → **Decision (reconciles [[plan#07 · `f-hub-capabilities` — Hub tools + MCP exposure|§07 t-2]]'s "human_approval where the design requires"):** the Hub write tools here are **direct, user-initiated actions** (the caller *is* the actor — a dev via MCP, or the user via the UI) → **no approval gate.** The §3.6 gate is on **sidekick proposals**, which is `f-sidekick`'s concern: the sidekick proposes → human approves (its own gated flow, mirroring `human-approval.ts`) → the approved action calls these same write capabilities. **This removes the resumable-approval build from this feature.**
- **Safety config lives on the seeded `AiCapability` row, not the class** (`orchestration-agents.prisma`): `rateLimit`, `requiresApproval`, `approvalTimeoutMs`, `isIdempotent`, `quarantineState`. The class declares only `processesPii` + `redactProvenance`. → **Decision:** free-text write tools (`create-task` title/description, `add-backlog`) set **`processesPii = true` + override `redactProvenance`** (registration *throws* otherwise). Any per-tool rate limit is a seed-row value (admin-tunable live), not code.
- **Audit via `logAdminAction`** (`lib/orchestration/audit/admin-audit-logger.ts`) — callable from `execute`, fire-and-forget (never throws), auto-redacts secrets. Precedent: `built-in/send-message-to-channel.ts`. → **Decision:** the write tools log `action: 'task.create' | 'task.claim' | 'feature.help_wanted' | …`, `entityType: 'app_task'|'app_feature'`, `entityId`, `userId: context.userId`, a `changes`/`metadata` payload. Reads don't audit.
- **Dependency-edge acyclicity is a shared write-path validator** (carried finding — [[f-data-model]] t-1 `/code-review`). `@@unique` blocks only duplicate edges; a self-loop (`a==b`) or multi-node cycle breaks topological `next-task`/`planOrder`. A DB `CHECK(a<>b)` catches only the self-loop base case. → **Decision:** build a **pure `assertAcyclic(edges, newEdge)` validator** (`lib/projects/dependency-graph.ts`) that rejects self-loops **and** cycles; `create-task` (task deps) calls it. The **same validator is reused** by `persist-features` ([[f-intake]]) and `propose-dependencies` ([[f-sidekick]]) — the other two edge-creating flows. This is the load-bearing enforcement.
- **The two read-path findings apply to `next-task`/reads** (carried — [[f-data-model]] t-2/t-3). Effective status must treat a `claimed` task with `claimedByUserId = null` as **not** claimed (return it to the pullable pool); and every nullable `Hub→user` ref (`ownerUserId`, `claimedByUserId`, lead) must be rendered/returned as "unassigned", never dereferenced. `next-task` computes effective status and must honour both.

**Tier / seam hypotheses (confirmed):** pure leaf-app. Fills `lib/app/capabilities.ts` (seam), adds a `lib/projects/capabilities/` domain module + `lib/projects/dependency-graph.ts`, seeds `AiCapability`/`McpExposedTool` rows via the (to-confirm) built-in seed path. **Zero core→fork seams, zero upstream asks.** No platform-file edit anticipated (the seam fill + seed are designed fork uses). Watch: if the built-in `AiCapability` seed path turns out to be a core-only mechanism with no fork hook, that's a possible small upstream ask (B17) — confirm early in t-1.

## Test strategy

Capability logic is **vitest-shaped** (mock `@/lib/db/client`; call `execute(args, context)` directly with a synthetic `CapabilityContext` — the `built-in/user-memory.test.ts` pattern). No live DB (B9). The load-bearing tests:
- **`next-task` matrix** — owned-vs-help-wanted, PR-blocked skip (a dep task `in_pr`/not `merged` blocks), membership scoping (non-member's projects invisible), null-claimant task treated as pullable, empty result. This is the algorithm; test it hard (B15).
- **`assertAcyclic` matrix** — self-loop (`a→a`), 2-node cycle (`a→b→a`), longer cycle, valid DAG edge, duplicate edge. Pure function, exhaustively unit-tested.
- **`claim-task` collision** — no overlap, overlapping open claim (soft warning, still claims), released claim (no warning), null/erased claimant (no wedge).
- **membership guard** — each capability with `context.userId = null` → `no_user_context` error; non-member → `forbidden`/`not_found` (never leaks existence).
- **No smoke** — no DB-referential effect beyond queries (contrast f-data-model). If the seed path proves subtle, a `scripts/app/smoke/*` that seeds + dispatches once against the dev DB may be added in t-1, decided at build.
