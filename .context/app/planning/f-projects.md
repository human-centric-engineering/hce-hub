# f-projects ‚Äî projects list + project-view scaffold

*Feature 08 on [[plan]]. Binding design: [[v1-requirements#13.1 Hub UI ‚Äî user-facing working surface|¬ß13.1]] (the user-facing working surface), design handoff "1. Hub home / Projects". Depends on [[f-shell]] (the `(hub)` shell it mounts into ‚Äî shipped) and [[f-project-admin]] (real projects to render ‚Äî shipped). **Unblocks [[plan#09 ¬∑ `f-plan-view` ‚Äî feature-level Plan view|f-plan-view ¬ß09]] + [[plan#10 ¬∑ `f-board-view` ‚Äî Board (Kanban)|f-board-view ¬ß10]].***

## Intent

The **first user-facing Hub module**: the membership-scoped project index at `/projects` and the project-view **container** the Plan/Board views mount into (¬ß09/¬ß10). It's the *consumer* mirror of [[f-project-admin]] ‚Äî where the admin surface *writes* `ProjectMember` rows for all projects, this surface lets an ordinary member *see and open only their own*, funnelled through [[f-access]]'s `lib/projects/access.ts` (a non-member gets 404, never 403). Plus the owner-intent **sample-data seed** that materialises the Hub's *own* v1 build plan into real rows, so the boards render something real and drivable from day one.

## Reconciliation with current repo reality   (required ‚Äî done first)

Verified against `main` after f-project-admin shipped, 2026-07-15 (recon over `lib/projects/access.ts`, `app/(hub)/**`, `lib/app/hub-modules.ts`, `prisma/seeds/app/**` + `prisma/runner.ts`, `prisma/schema/app.prisma`, the f-project-admin consumer/serverFetch pattern). Each finding is a decision.

- **No consumer projects REST API exists yet ‚Äî f-projects builds it (API-first).** `app/api/v1/**` has only the *admin* `projects` routes (f-project-admin) + the Hub *capabilities* (f-hub-capabilities, agent-facing). There is **no** `/api/v1/projects` for the UI. ‚Üí **Decision:** build a consumer read API ‚Äî `GET /api/v1/projects` (list) + `GET /api/v1/projects/[id]` (detail) ‚Äî `withAuth` (session **or** `sk_` key ‚Üí the funnel applies uniformly, per [[f-access]]), membership-scoped via the funnel; the `(hub)` pages fetch it with `serverFetch` (mirrors f-project-admin's admin pages + satisfies the API-first / no-direct-Prisma-in-pages rules, CLAUDE.md). *Not* a direct `listAccessibleProjects()` call from an RSC.
- **The consumer funnel is ready ‚Äî use it, never hand-roll membership.** `lib/projects/access.ts` (f-access) gives `listAccessibleProjects(userId) ‚Üí Project[]` (member projects, newest first), `getAccessibleProject(userId, projectId, need) ‚Üí Project` (**deny ‚â° 404**, the anti-enumeration shape), and `accessibleProjectIds(userId) ‚Üí string[]` (the child-query scoping primitive). ‚Üí **Decision:** the list route returns `listAccessibleProjects` enriched (lead + member + feature/task counts); the detail route uses `getAccessibleProject` (404 for a non-member/unknown id). The enrichment (lead identity, member stack, counts) is a **separate `user` lookup + `_count`** ‚Äî the hand-FK has no Prisma relation (same as f-project-admin's admin service).
- **Single enriched endpoint, no N+1 (CLAUDE.md).** The card grid needs name/platform/lead/member-count/feature-&-task-counts per row. ‚Üí **Decision:** the list endpoint returns all of it inline (`_count` + one batched `user` lookup for leads); the detail endpoint returns the project header + members + feature/task **counts** ‚Äî **not** the full feature/task lists. Rendering features/tasks is ¬ß09/¬ß10; building their read endpoints here would ship unconsumed surface. ‚Üí **the Plan/Board tabs are empty scaffolds in ¬ß08**; ¬ß09/¬ß10 add the feature/task loaders they render.
- **Pages mount in the shipped `(hub)` shell; the Projects placeholder is replaced.** `app/(hub)/projects/page.tsx` is f-shell's placeholder; the Projects module is already registered (`lib/app/hub-modules.ts` ‚Üí `/projects`). ‚Üí **Decision:** replace `page.tsx` with the real list; add `app/(hub)/projects/[id]/page.tsx` (the container). The group-level auth guard + `/projects` edge gate ([[f-access]]) already protect these ‚Äî no new guard. **No HB2** ‚Äî `(hub)/projects/page.tsx` is fork-owned (f-shell created it), so replacing it is not a platform divergence.
- **Sample-data = TWO seed units (owner-decided 2026-07-15), because a member needs a real `user` row.** `ProjectMember.userId` is a hand-FK to `user` with a real DB constraint ‚Äî you **cannot** attach a member whose user doesn't exist, so previewing multi-member UI on a solo dev DB requires *fabricating* a user, which must never reach prod. Seeds run on **every** `db:seed` (dev, CI, prod ‚Äî prod already runs the `001‚Äì005` capability seeds). ‚Üí **Decision ‚Äî split:**
  - **`006-sample-plan.ts` ‚Äî all envs, prod-safe.** Materialises *this plan* into `app_project`/`app_feature`/`app_task`/deps as the **real** HCE Hub coordination board (it's the board they'll drive). **Idempotency:** explicit **stable string ids** (`id: 'seed-hub-project'`, `id: 'seed-feat-<slug>'`, `id: 'seed-task-<slug>-<n>'`) + `upsert` (the models take any `String @id`). **Membership:** resolve builder users **by real email** (Simon ‚Üí lead; John auto-added iff a `john@‚Ä¶` user exists) ‚Äî **no fabricated users**; on a solo dev DB you're simply the sole member. Feature/Task **statuses** map from the plan's `shipped`/`in flight`/`planning` + the task board vocabulary.
  - **`007-demo-collaborators.ts` ‚Äî dev-only, GATED (`NODE_ENV !== 'production'`, early-return).** Creates 1‚Äì2 clearly-labelled demo users (e.g. "Ada Lovelace"/"Grace Hopper"), adds them to the sample project, leaves one task claimed + one owner null ‚Äî so a solo dev DB shows the **member stack / avatars / "former-member" / "unassigned"** rendering. **Never runs in prod ‚Üí the real user list stays honest.** Idempotent (stable demo-user ids).
- **The Plan‚áÑBoard tab is a linkable route (design: "view is part of the route").** ‚Üí **Decision (build-time reconciliation, √Ý la [[f-shell]]'s routing choice):** scaffold the container's tab control as a **linkable route** ‚Äî either a `?view=plan|board` search param (simplest; default `plan`) or a nested segment. Pick the cleaner one at build; ¬ß08 ships the container + the tab *control* (the tab *content* is ¬ß09/¬ß10). Recommend `?view=` unless a nested segment reads cleaner with the shell.

**Tier / seam hypotheses (confirmed):** pure leaf-app, **zero platform edits, zero upstream asks**. New fork-owned consumer API routes (`app/api/v1/projects/**` ‚Äî rate-limit automatic via `proxy.ts` session-user cap), new `(hub)/projects` pages + components, a new seed unit. `lib/projects/access.ts` is the **live** dependency (its first consumer-API caller ‚Äî [[feature-plan-authoring-guide]] B27: put the load-bearing test on the 404-not-403 wiring). No HB2 (no fork-owned default-test asserts these).
