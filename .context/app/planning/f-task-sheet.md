# f-task-sheet — task detail side sheet

*Feature 11 on [[plan]]. Binding design: [[v1-requirements#5. The PR is the natural unit|§5]] (the PR as the unit; files "declared, not enforced"; pull-not-push claim), [[v1-requirements#3. Human-centric principles (binding)|§3.6]] (approval — the sheet's actions are human-initiated, so ungated), [[v1-requirements#13.5 Tone, feel, and anti-patterns|§13.5]] (quiet, no alarms), design handoff "7. ## Intent

The **task sheet** — full task detail *without losing context*: a **deep-linkable side sheet** that slides in from the right over a light scrim, opened from a Plan row or a Board card (later the brief + sidekick), shareable by URL, and — the specific design requirement — **repositioning to the LEFT of the sidekick when the sidekick is open** (`right: 392px`) so a task and the sidekick are readable together. It renders the task's real state (status, claimer, PR, description, files-in-scope, dependency graph) and an **action row** wired to the Hub's ungated write capabilities (Claim / Open PR / Open in Claude Code / Ask sidekick). It closes the interaction loop the Plan and Board opened: both already carry `// opening the sheet is §11` stubs on their task rows/cards.

## Reconciliation with current repo reality   (required — done first)

Verified against `main` after f-refs shipped, 2026-07-16 (recon over `lib/projects/{plan,board,task-status,collision,access,user-refs}.ts`, `lib/projects/capabilities/claim-task.ts`, `components/hub/{hub-shell,sidekick-column}.tsx`, `components/hub/projects/{project-view,plan/task-row,board/task-card}.tsx`, `app/(hub)/{layout,projects/[id]/page}.tsx`, `prisma/schema/app.prisma` `Task`/`TaskDependency`/`TaskClaim`, `design/task-sheet.jsx` + README §7). Each finding is a decision.

- **No single-task read exists → build `getTaskDetail(userId, projectId, taskId)` + `GET /api/v1/projects/[id]/tasks/[taskId]`.** `/plan` and `/board` load whole graphs; the sheet needs one task's detail (description, `filesScope`, effective status, claimer, PR, and its `blocked-by`/`blocks` edges). → **Decision:** `lib/projects/task-detail.ts` `getTaskDetail(...)` (`withAuth`, `parseCuidParam` on both ids), routed through `getAccessibleProject` → **404-not-403** (reuse the funnel; a task outside a member's project ≡ not found), and re-check the task belongs to the resolved project (no cross-project id-swap). Reuse `computeEffectiveStatus` (Plan/Board never diverge), `fetchUsers`, `sanitizeUrl` on `prUrl`. Returns the detail **shaped server-side**; the client stays dumb.
- **`Task` has the real content the sheet needs — `description` + `filesScope`.** `Task.description String?` and `Task.filesScope String[]` are shipped (f-data-model). → **Decision:** description + "files in scope" render **real data** (files as the soft "declared, not enforced" list, §5) — no mock. Empty `filesScope`/`description` (the seed sets neither) render a quiet empty state.
- **`TaskDependency` is a real relation but EMPTY in v1 → the dependency graph renders honest empty states.** Task deps are a claim-and-plan-time concept; none are seeded and no flow creates them until `f-sidekick propose-dependencies` (§12) / `f-intake persist-features` (§13) — owner-confirmed deferred. → **Decision:** build the two-column `blocked by` / `blocks` graph against the real `TaskDependency` relation (`dependencies`/`dependents`), so it lights up automatically when deps exist; on the v1 seed every task shows "none — ready to pull" / "nothing waiting". Each row is click-to-jump (rewrites `?task=`). **No mocked edges.**
- **`claim_task` is a *capability*, not an HTTP route → add a thin consumer endpoint that reuses its core (API-first).** The action row's **Claim** needs an HTTP path; today `claim_task` is only reachable via MCP/chat tool-loop. CLAUDE.md: "every capability must be API-accessible". → **Decision:** `POST /api/v1/projects/[id]/tasks/[taskId]/claim` (`withAuth`, funnel-scoped, **per-flow sub-cap** — a claim is a write) that reuses the capability's claim logic. Extract the claim core into a shared `claimTask()` service in `lib/projects/` that **both** the capability and the route call (don't duplicate the soft-collision/`TaskClaim`/null-claimant logic) — mirrors how reads share `computeEffectiveStatus`. Returns the soft `warnings[]` so the sheet can show "John touched `/api/auth` 2h ago" without a hard lock (§5). Open-PR / Ask-sidekick are non-mutating (a link, a panel open); **Open in Claude Code** is a deep-link (below).
- **The sheet is a CLIENT overlay reading `?task=` — mount it once, above the tabs, so it survives Plan⇄Board without a page refetch.** The page is an RSC; the design requires the sliding sheet to **preserve the underlying surface state** (not re-render Plan/Board). → **Decision:** a client `<TaskSheetHost>` mounted in `project-view.tsx` (inside the project route, above both tabs) reads `useSearchParams()` for `?task=`, and on open **fetches detail client-side** from the t-1 endpoint (loading skeleton → content). `?task=` **composes with the existing `?view=`** param (both are URL state; `?view=` stays server-driven for the tab, `?task=` is client-driven for the overlay). Prefer `?task=` over a parallel/intercepting route: shareable, survives refresh, no route-tree restructure, and it layers cleanly on the `?view=` scaffold f-projects already shipped. Esc + scrim close (clear `?task`), copy-link copies `window.location.href`.
- **Reposition-beside-sidekick needs the shell's `sidekickOpen` state → lift it to a small fork-owned context.** `hub-shell.tsx` owns `sidekickOpen` locally (`useState`); the sheet (a main-column overlay) must read it to set `right: 392px` vs `right: 0`. → **Decision:** a fork-owned `SidekickContext` provided by `hub-shell.tsx` exposing `{ open }` (later `{ open, toggle }` for "Ask sidekick" to open it), consumed by `<TaskSheetHost>`. Fork-owned `components/hub/**` only — **no platform edit**. The sidekick **column** is still a §12 placeholder, but the shell already toggles it, so building the reposition now means §12 inherits it working. On narrow viewports the sheet narrows to keep both visible (design §7).
- **Open triggers: wire the Plan row + Board card (both display-only today, comments already point here).** `plan/task-row.tsx` and `board/task-card.tsx` each say "clicking opens the task sheet in f-task-sheet §11 (not wired here)". → **Decision:** make each task row/card push `?task={id}` (client nav, no scroll reset) — the sheet host reacts. Brief (§15) + sidekick (§12) triggers wire when those surfaces land (the host is surface-agnostic; it only needs `?task=`).
- **"Open in Claude Code" MCP deep-link — homed here from [[f-hub-capabilities]] t-6.** §07's split-note parked the deep-link for this feature; the MCP connection guide is `.context/app/mcp-claude-code.md`. → **Decision:** build the deep-link as a small pure helper (`lib/projects/claude-code-link.ts`) that composes the documented Claude-Code/MCP URL for a task; **`sanitizeUrl` it** before rendering (same safe-by-construction discipline as `prUrl`). If the exact deep-link scheme isn't yet nailed down, ship it behind the honest label and confirm the URL shape at build (owner decision below).
- **[carried-in — [[f-board-view]] §10 `/code-review`] Fix the Board's sticky header during this feature's scroll/overflow work (→ t-2).** `BoardHeader` is `sticky top-0` nested inside the `overflow-x-auto` wrapper (`board-view.tsx`), which resolves to a both-axes scroll container so the header may scroll away vertically. The sheet's own scroll/overflow layout is worked in t-2. → **Decision:** discharge here — either drop the sticky or restructure so vertical page-scroll and the horizontal-scroll container don't share an ancestor.
- **[carried-in — [[f-plan-view]] §09 `/code-review`] Do the disclosure/interactive-control a11y pass (→ t-3).** The Plan feature-row expand `<button>` (`plan/feature-row.tsx`) has no concise `aria-label` (its accessible name is the whole head text) and no `aria-controls`. → **Decision:** discharge with t-3's interactive-control work — `aria-label={`Toggle tasks for ${feature.title}`}` + `aria-controls`→an `id` on the expanded region; **apply the same pattern to the sheet's own controls** (`role="dialog"` + `aria-label`, labelled close/copy buttons, focus-trap + return-focus-on-close, Esc).

**Tier / seam hypotheses (confirmed):** pure leaf-app, **zero platform edits, zero upstream asks anticipated**. New fork-owned: `GET /api/v1/projects/[id]/tasks/[taskId]` (read) + `POST …/claim` (per-flow rate-limited write), `lib/projects/{task-detail,claude-code-link}.ts` + the shared `claimTask()` service, `components/hub/projects/task-sheet/**`, a fork-owned `SidekickContext` in `hub-shell.tsx`, and open-trigger edits to `plan/task-row.tsx` + `board/task-card.tsx`.
