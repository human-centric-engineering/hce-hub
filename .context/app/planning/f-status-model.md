# f-status-model — readiness-derived feature + task status

*Feature 20 on [[plan]] — a **corrective** feature (the way [[f-refs]] §16 was an inserted correction), surfaced browser-validating the §19 cutover: the Hub's status labels don't reflect reality. Depends on [[f-selfhost-cutover]] §19 t-2 (the cutover module whose statuses this re-authors — the cutover data is the live test-bed). **Sequenced before the §19 close-out** (owner) — no point freezing a system-of-record that mislabels its own statuses. Corrects the status handling in [[f-plan-view]], [[f-board-view]], [[f-feature-planning]].*

## Intent

Make status **reflect reality**, derived — not a hand-typed label — at both levels:

- **Task statuses collapse to `claimed | active | merged`** (owner, 2026-07-23). A task is *born* **claimed** (owned by the feature's owner) the moment you plan a claimed feature — there is no `backlog`/`available` limbo, because a task doesn't exist until a feature is claimed *and* planned. **active** = being worked; **merged** = done. "blocked" stays a *derived* overlay (a claimed task with an unmerged dependency). We can get more granular (in-PR, in-review) when [[plan#14 · `f-github-sync`|f-github-sync]] lands; three is enough now.
- **Task-level pulling goes away.** "Claim features, not tasks": you pull a **feature** ([[f-feature-planning]]'s `claim_feature`, shipped), and its tasks are yours. So the Board's *Available* column, the task-sheet **Claim** button, and the `claim_task` capability/route are retired; the sheet's action becomes **Start** (claimed → active) then **Complete** (active → merged). The assignee-vs-claimer split collapses — a task's owner is the feature's owner.
- **Feature status is derived, not labelled `planning`.** An *unclaimed* (not-started) feature is **available** (all its feature dependencies are shipped) or **blocked · waiting on `<dep>`** (a dependency isn't shipped) — the same readiness derivation the task model uses. Once claimed → `in_flight`; done → `shipped`. The static "planning" pill disappears.
- **Plan-view presentation fixes (folded in, owner 2026-07-23; description-split re-homed to §21, 2026-07-24).** Browser-validating the same view surfaced three defects on the *presentation/identity* axis: (a) **stable feature number** — the leading number is the row's *position* in `planOrder()`, so §20 shows "16"; give features a project-wide stable `Feature.number` (via a new `Project.featureCounter`, backfilled from `createdAt` = the plan.md §N; mirrors `Task.number`/`taskCounter`), **displayed** while the list still **sorts** by `planOrder`; (b) **numerical task order** — the plan read orders tasks by `createdAt` (the backdated-import order), should be `number` asc; (c) **description split** — one raw `description` overflows the plan row *and* leaks literal markdown. **(a) rides t-2** (feature-readiness + identity); **(b) shipped in t-1** (t-36); **(c) MOVED to [[f-authoring-fidelity]] §21 t-d** — it's the same "faithful detail" axis as the MCP authoring fix (summary/description split + markdown), not status/identity. A third t-2 fix joined at build time: the **plan-view auto-expand** opened the first non-shipped feature *by dependency depth* (so §19 pre-empted the feature with live work) → prefer the feature holding an `active` task.

*(Out of scope — parked, owner 2026-07-23: **external / Sunrise-gap dependencies.** A feature blocked by a missing Sunrise primitive would flag "blocked · Sunrise `<cap>` gap" and follow fix-on-fork → raise-issue. But **nothing in v1 is blocked by a Sunrise gap** — the plan's headline finding is "zero core→fork seams; the Hub builds entirely on existing Sunrise seams", and scheduling/agents/webhooks/workflows all exist. No consumer → don't build it (ship-nothing-unused). Design captured in [[self-hosting]]; build on the first real gap.)*

- **The stored-enum change is *unconsumed-until-complete* — it must land as one vertical.** Every writer (`plan-feature`/`create-task`/`add-backlog`), reader (`board`/`plan`/`next-task`/`task-detail`/`feature-detail`), the Board columns, the pills/types, and the cutover data reference the old values; changing the enum without them breaks the app. → **Decision:** t-1 is **one PR** (HB3 — a stored-enum migration + its whole vertical is one PR even when large), split from the feature-readiness derivation (t-2, no schema change).
- **`plan_feature` creates tasks `status: 'available'`; `create_task` → `'available'`; `add_backlog` → `'backlog'`.** → **Decision:** `plan_feature` creates tasks **`claimed`** (owned by the feature owner — the born-claimed cascade); `create_task` → `claimed`; **retire `add_backlog`** (the `backlog` value is gone; a not-yet-planned feature holds *indicative* tasks, not backlog `Task` rows — self-hosting §2). Its seed/capability/registration/tests go with it.
- **Task-level claiming is a whole surface: `claim_task` capability (`005-claim-task` seed) + `claim-task-service.ts` (`claimTask` writes `claimed` + a `TaskClaim` + journals `task_claimed`) + `POST …/tasks/[taskId]/claim` + the sheet Claim button (`canClaim = status==='available'`).** → **Decision:** **retire `claim_task`** (capability/seed/registration/barrel export) and **repurpose the service + route as `startTask` / `completeTask`** (`claimed → active`, `active → merged`) driving a contextual sheet action (**Start** when claimed, **Complete** when active). Reuse the **`task_claimed`** `ProjectEventKind` for Start (semantically "actively taken") + **`task_merged`** for Complete (both already in the enum — **no `ProjectEventKind` migration**). The `TaskClaim` row is **repurposed as the "active work" record** written on Start, so soft-collision + history survive the loss of task-claiming.
- **Board columns are hardcoded in 4 places** (`board.ts` `BOARD_COLUMNS` const + `columnTotals` literal; `board/types.ts` `BoardColumn`; `board/presentation.ts` `COLUMN_META` + `groupByColumn` + `BOARD_GRID` `repeat(5)`); routing maps `effective==='blocked' → 'backlog'`. → **Decision:** columns become **`claimed | active | merged`** (grid `repeat(3)`); a **blocked** task shows in its `claimed` column with the blocked treatment (no separate column — it's a claimed task that can't start yet); collision derives from **overlap among `active` tasks' `filesScope`** (+ the repurposed `TaskClaim`). Move all four spots together (landmine — desync risk).
- **Presentation: `TASK_TONE` maps all 6 effective values; `FEATURE_TONE`/`plan-summary` map `planning`; `StatusPill` tone is a `--signal-*` token.** → **Decision:** `TASK_TONE` → `{claimed, active, merged, blocked}`; add a **`--signal-active`** token (+ `-bg`) to the fork-owned `app/brand-theme.css` (else the pill renders an undefined var); `FEATURE_TONE`/`plan-summary` lose `planning`, gain derived `available`/`blocked` (t-2). Client type mirrors (`plan/types.ts` `TaskEffectiveStatus` + `FeatureStatus`, `board/types.ts`, `task-sheet/types.ts`, `feature-view/types.ts`) track the new sets.
- **Feature status is returned raw (`plan.ts:158-166` `PlanFeatureView.status: FeatureStatus`); the dep select fetches only `dependsOnFeatureId` (no status).** → **Decision (t-2):** derive an **effective feature status** — for a not-`in_flight`/`shipped` feature, `available` if **every** dependency feature is `shipped`, else `blocked` carrying the **unshipped dep slug(s)** for a "waiting on `<dep>`" reason. Add each dependency's `status` (+ slug) to the select. Mirror in `feature-detail.ts`. Keep `plan-order.ts` `STATUS_BAND` on the stored value (ordering unchanged). Stored `FeatureStatus.blocked` is **reserved** (the parked external model), unused by the derivation.

- **Plan-view presentation (folded in; description-split re-homed to §21).** `plan.ts` orders tasks `createdAt: 'asc'` (fixed in t-1); the feature's leading number is `plan-view.tsx` `ordinal={i + 1}` (positional, unstored); no `Project.featureCounter`. → **Decisions:** (t-1, shipped) task read `orderBy: { number: 'asc' }` with nulls last + `createdAt` tiebreak. **(t-2)** one **additive** migration — `Feature.number Int?`, `Project.featureCounter Int @default(0)`: backfill `Feature.number` per project from `createdAt` rank (= plan.md §N) and set `featureCounter` to the max; `create_feature` assigns the next `featureCounter` (mirror `taskCounter`); render the stable number in `feature-row` + `feature-view`, keep the sort on `planOrder`; the transfer round-trip + cutover snapshot carry `number`/`featureCounter`. **The `Feature.summary` split + markdown rendering MOVED to [[f-authoring-fidelity]] §21 t-d** (same faithful-detail axis as the MCP authoring fix) — no `summary` column here.
