# f-feature-planning — feature lifecycle + indicative vs planned tasks

*Feature 18 on [[plan]] (the second of the **self-hosting cluster** §17→§18→§19). Binding design: [[self-hosting#2 · Feature-planning model (threads 2, 3, 4)|self-hosting §2–§4]] (the model, the verbs, the UI) + [[self-hosting#7 · Plan-doc coverage — the completed plans vs the model (assessment 2026-07-17)|§7]] (the `Task.doneWhen` carry). Depends on [[f-journal]] (every verb emits a `ProjectEvent` — shipped) and [[f-plan-view]] (the Plan tree the indicative/planned rendering extends + the `getProjectPlan` read whose §09 progress-calc carry resolves here — shipped). **Unblocks [[plan#19 · `f-selfhost-cutover`|f-selfhost-cutover §19]]** (the import seed materialises features through this model) and, from the cutover, the whole studio manages delivery through these verbs. Built via the **current GitHub flow** — can't dogfood the Hub to build its own self-hoster.*

## Intent

The model + verbs for **how we actually work**: you **claim features, not tasks** — tasks aren't defined until a feature is *planned*. This feature makes that real over MCP:

- **Indicative tasks** — a lightweight, ordered `IndicativeTask` list on a feature (the high-level sketch, the plan's `- t —` bullets): not claimable, no number, no PR.
- **Planned tasks** — the real `Task` rows, created at **plan time** (assignee-defaulted to the feature owner, `number` via the counter), replacing the indicative sketch.
- An orthogonal **`Feature.planningStage`** (`indicative | planned`) that drives the Plan UI's at-a-glance "planned & ready vs still high-level" — distinct from `status` (the ownership/progress axis).
- First-class **`Feature.doneWhen`** + a populated **`description`** + **`references`** cross-refs, and per-task **`Task.doneWhen`** (the acceptance contract — the [[self-hosting#7 · Plan-doc coverage — the completed plans vs the model (assessment 2026-07-17)|§7 coverage carry]]).
- The lifecycle verbs `create_feature` / `claim_feature` / `plan_feature` / `ship_feature`, each membership-scoped + audited + **journalled** (they emit the `feature_*` `ProjectEventKind`s f-journal defined but didn't yet emit).
- **`assertAcyclic`** — the B26 dependency-cycle guard, finally landing at its first real user (`plan_feature`'s task batch can reference itself into a cycle).
- The UI: the Plan view renders indicative vs planned distinctly, and a **`?feature=` feature-detail side-sheet** (the `?task=` sibling) shows description / done-when / refs / tasks / the feature-scoped journal.

## Reconciliation with current repo reality   (required — done first)

Verified against `main` after f-journal shipped (#64), 2026-07-17 (recon over `prisma/schema/app.prisma` `Feature`/`FeatureDependency`/`Task`/`TaskDependency`, `lib/projects/{access,plan,plan-order,task-status,project-event,journal,claim-task-service}.ts`, `lib/projects/capabilities/{create-task,claim-task,flag-help-wanted}.ts`, `lib/app/{capabilities,db-drift}.ts`, `prisma/seeds/app/*`, `components/hub/projects/{plan/**,task-sheet/**,project-view}.tsx`). Each finding is a decision.

- **`Feature` already has `description @db.Text` + `slug` (per-project unique) + `FeatureStatus` + `FeatureDependency`; it lacks `doneWhen`/`references`/`planningStage`.** → **Decision:** add `doneWhen String? @db.Text`, `references Json?` (a list of `{ label, target }`), `planningStage FeaturePlanningStage @default(indicative)` (+ new enum `{ indicative planned }`). `description` is populated by the verbs (no schema change). `slug` reuses the shipped f-refs `@@unique([projectId, slug])` (authored, collision → error).
- **No `IndicativeTask` model exists.** → **Decision:** add it per the design (`id`, `featureId`, `order Int`, `text @db.Text`, `@@map("app_indicative_task")`), with an **intra-Hub `@relation` to Feature, `onDelete: Cascade`** (the sketch dies with the feature — no user data, no satellite FK). Add `indicativeTasks IndicativeTask[]` to `Feature`.
- **`Task` has `number`/`filesScope`/`claimedByUserId` but no `assigneeUserId`/`doneWhen`.** → **Decision:** add `assigneeUserId String?` (**hand-FK → `"user"` SET NULL** + drift probe — a retained-work satellite FK, exactly like `claimedByUserId`; "this is yours to do", defaults to the feature owner at plan time, freely reassignable, kept **distinct** from the pull-`claim`) and `doneWhen String? @db.Text` (per-task acceptance contract). **One additive migration**, B13-strip watch (the f-journal footgun *will* recur), `db:drift-check` green.
- **`plan_feature`'s task batch is the first flow that can create a *cycle* → `assertAcyclic` lands here (B26).** `create_task`/`add_backlog` only add a new leaf (outgoing edges only — can't cycle, HB4); but `plan_feature` creates **N new tasks that may depend on each other** (ref `t2` → `t1` → `t2`) as well as on existing tasks. → **Decision:** build `assertAcyclic(edges)` as a **pure guard** in `lib/projects/dependency-graph.ts` (generic `{ from, to }[]` → detect a self-loop / any cycle via DFS or Kahn's; throws a typed `DependencyCycleError`), exhaustively tested (self-loop, 2-cycle, N-cycle, diamond-DAG passes, disconnected). `plan_feature` validates the combined graph (new task refs + existing task ids) **before** the write. `create_feature`'s `dependsOn` is **create-time feature edges from a brand-new node → no cycle possible**, so it does *not* call `assertAcyclic` (same reasoning as `create_task`; a standalone feature dep-edit verb — which *would* need it — is out of scope, deferred).
- **`plan_feature` creates real `Task` rows → reuse the f-refs counter-bump invariant.** `create_task` assigns `number` via `tx.project.update({ taskCounter: { increment: 1 } })` inside its tx (unique by construction, no DB backstop). → **Decision:** `plan_feature` bumps the counter **once per task** in its transaction; sets `assigneeUserId = feature owner`, `doneWhen`, `filesScope`, `status: 'available'`; creates `TaskDependency` rows (resolving batch-local refs → created ids); **deletes the feature's `IndicativeTask` rows** (planning *replaces* the sketch — design §2); sets `planningStage → planned`; emits `feature_planned` + `task_created`×N (reusing `recordProjectEvent(tx, …)`, so the events are atomic with the plan).
- **The lifecycle verbs follow the shipped capability recipe + emit the `feature_*` kinds f-journal pre-defined.** `BaseCapability` (slug + `functionDefinition` + Zod `schema` + `execute`), funnel-scoped, `logAdminAction`, seeded active `AiCapability` + `McpExposedTool` + class↔seed parity, registered in `lib/app/capabilities.ts`. Free-text carriers (`create_feature` title/description/doneWhen, `plan_feature` task titles/doneWhen, `ship_feature` summary) set `processesPii=true` + `redactProvenance`. → **Decision:** `create_feature` (author at `planningStage: indicative`, emit `feature_created`), `claim_feature` (owner = caller + `status → in_flight`, emit `feature_claimed`, pull-not-push soft-warn if already owned), `plan_feature` (owner-tier, above; emit `feature_planned` + `task_created`×N), `ship_feature` (owner-tier, `status → shipped`, emit `feature_shipped` with the summary as `body`; **soft-warn** on unmerged tasks — never a hard block, §5).
- **The §09 progress-calc carry resolves here.** `getProjectPlan` (`lib/projects/plan.ts`) computes `progress.live` off **stored** task status (`status !== 'merged' && !== 'backlog'`) while task rows render **effective** status — they only diverge once a task can be effectively `blocked`, which needs a `TaskDependency`, and **`plan_feature` is the first flow that creates task deps**. → **Decision:** switch `progress.live` (and the blocked count) to `computeEffectiveStatus`, so a dep-blocked task reads consistently in its row *and* its feature's progress. (Carried from [[f-plan-view]] §09 `/code-review`.)
- **The Plan read + UI must carry `planningStage` + the indicative list.** → **Decision:** extend `getProjectPlan` to select `planningStage` + `indicativeTasks` (ordered); the Plan `feature-row` renders an **indicative** feature's sketch *muted/dashed, no status pills* and a **planned** feature's real `Task` rows as today, with a planned/indicative affordance on the row. No new endpoint for the Plan (extend the shipped `/plan`).
- **The feature-detail surface reuses the shipped `?task=` overlay pattern.** The task sheet is a client `?task=` overlay via the History API (`task-sheet-host.tsx`), repositioning beside the sidekick. → **Decision:** a **`?feature=` sibling** — `getFeatureDetail(userId, projectId, featureId)` (funnel → 404-not-403) + `GET /api/v1/projects/[id]/features/[featureId]` returning description / doneWhen / references / owner / dependsOn / the indicative-or-planned task list; a `<FeatureSheet>` overlay mounted beside the task sheet, whose **activity section reuses `getProjectEvents({ featureId })`** (the feature-scoped journal — decisions + activity). Plan rows + the feature row open it. (Lean side-sheet over a Plan-row expansion — reuses the shipped, owner-validated pattern; owner decision below.)
